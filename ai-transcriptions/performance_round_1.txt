     "Similar to the composite index, since search queries are only

  3. PRRT_kwDOS8vrP86KEcFC  blog/api.py:34
     "Instead of manually parsing and validating query parameters from
      `request.GET`, it is highly recommended to leverage Django Ninja's
      native query parameter handling. Why: (1) OpenAPI/Swagger
      documentation auto-generates from the signature — `limit` and

     Suggested: declare `limit: int = 50, offset: int = 0` directly in

All three are valid:

  1. The partial index is the strictly tighter match for the queries we
     run, and the savings (smaller index, no write amp on drafts, better
     buffer cache hit rate) are real even if small on this dataset
     (90% of rows are published). On a production dataset with a more
     typical draft ratio the saving compounds.

     reading /api/docs wouldn't know `limit`/`offset` exist. And the
     422 behavior on bad input is the standard HTTP contract — silently
     falling back to defaults is a footgun.
