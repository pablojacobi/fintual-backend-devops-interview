Sí, había un mecanismo de seed antes de mis cambios. La respuesta corta:

- **Cómo se corría (sin mis cambios):**
  ```sh

  ```
  Es decir, después de `migrate`, lo invocabas manualmente desde el `manage.py` de Django.

- **Qué es:** un management command de Django en `blog/management/commands/seed.py`. No es un script externo ni una fixture; es código Python que se ejecuta dentro de la app Django, con acceso a los modelos, ORM, settings, etc.

- **Qué siembra** (constantes arriba del archivo `seed.py`):
  - `NUM_USERS = 1000`
  - `NUM_TAGS = 50`
  - `NUM_POSTS = 100_000` (~100k)
  - `NUM_COMMENTS = 500_000` (~500k)
  - 3 tags promedio por post
  - Se hace en `bulk_create` con `BATCH=1000` dentro de `transaction.atomic()`.

- **Idempotencia:** el comando chequea `User.objects.exists()` y, si ya hay datos, no hace nada salvo que le pases `--force`:
  ```sh
  uv run python manage.py seed --force
  ```

- **Tiempo:** el README avisa que toma "a few minutes" (en la práctica, con `bulk_create` por batches de 1000, son 1–3 min en una Mac razonable; más en CI).

- **Reproducibilidad:** usa `Faker.seed(42)` y `random.seed(42)`, así que la misma corrida produce el mismo dataset (útil para comparar performance antes/después).

- **Cosas intencionalmente feas que el seed te permite ejercitar** (lo que después el README te pide perfilar):
  - 100k posts con `view_count` random y mezcla "reciente" (180 días) vs "viejo" (3 años).
  - Distribución power-law de autores: 10 autores top escriben 30% de los posts.
  - Distribución long-tail: el top 1% de posts recibe 50% de los comments.
  - M2M `Post.tags` con mezcla sesgada hacia 5 "hot slugs" (`python`, `django`, `postgres`, `devops`, `sre`).

Esto último es por diseño: el seed pre-construye el sesgo que va a romper los índices "obvios" en el endpoint `/api/posts/by-tag/python` — lo vas a notar cuando lo perfiles.

**4. ¿Hot reload?**

Sí, hay hot reload. Funciona por dos razones que se complementan en el `Dockerfile.dev` y el `docker-compose.yml`:

1. **El comando del contenedor es `python manage.py runserver 0.0.0.0:8000`**, no `gunicorn` ni `uvicorn`. `runserver` de Django trae un watcher basado en `StatReloader` (usa `os.stat` o `inotify` según la plataforma) que reinicia el proceso cada vez que cambia un `.py`. Esto ya estaba así en el proyecto original — yo no lo cambié.

2. **El proyecto está bind-mounteado en el container:**
   ```yaml
   volumes:
     - .:/app
   ```
   Eso significa que el `/app` que ve Django adentro del container es el mismo árbol de archivos de tu Mac. Cuando guardas un archivo en tu editor, el evento filesystem se propaga al container y el `StatReloader` lo ve y reinicia.

**Caveats del hot reload que vas a encontrar en la práctica:**

- **No hay `watchfiles`/`watchdog` instalado** (Django 5.2 ya lo trae como opcional, pero el `pyproject.toml` no lo declara). Por defecto usa `StatReloader`, que sondea cada 1s. Es menos elegante que `inotify` pero funciona sin dependencias nativas.
- **Cambios en `pyproject.toml`/`uv.lock` no recargan** — `runserver` no observa esos. Hay que rebuildear (`docker compose up --build`).
- **Cambios en `core/settings.py` no se recargan del todo bien** — Django detecta el cambio, intenta reiniciar, y a veces falla porque el módulo `core.settings` ya está importado en el proceso padre. Workaround: tocar un `.py` de `core/` para forzar el reinicio.
- **Si añades una migración nueva** (`makemigrations`), tienes que aplicarla manualmente: `docker compose exec web python manage.py migrate`. `runserver` no lo hace solo.

Si quieres **mejor** hot reload (más rápido, sin polling), se puede añadir `watchfiles` como dev-dep y correr `watchfiles "python manage.py runserver 0.0.0.0:8000" blog core manage.py` en vez del `runserver` directo. Pero eso es un cambio adicional, no entra en el set mínimo.

**5. ¿Tiene sentido instalar dev-deps en el container de dev?**

Sí, totalmente. Lo cambié: `uv sync --frozen` sin `--no-dev`, así pytest/ruff quedan dentro del `.venv` del container y puedes:

```sh
docker compose exec web python -m pytest
docker compose exec web ruff check .
```

El costo es ~5–10 MB más en la imagen y un par de segundos extra en el primer build. En dev, trivial. El razonamiento de por qué lo había puesto `--no-dev` era "la imagen de runtime no necesita tests" — eso es válido para producción, pero en **`Dockerfile.dev`** es exactamente al revés: dev es el lugar donde sí quieres los tests y el linter corriendo adentro. Buen catch.

- La propuesta de Gemini `in ("1", "true", "yes", "on")` cambia el default a `"False"` ("default to `False` for security reasons in production-like environments"). **Eso rompe la regla de "no cambiar el comportamiento"** del PR: el código original tenía `DEBUG = True` hardcodeado. Si acepto el cambio, alguien que clone el repo y corra `docker compose up` sin tocar `.env` va a obtener `DEBUG=False` (que es más seguro, sí, pero es un cambio de comportamiento del default).
- **Compromiso que se ajusta a tu regla "no salirse del DX":** aceptar la lista de truthy values pero **mantener el default `"1"`**. Así:
  - Quien ya tenía `DJANGO_DEBUG=1` o no lo define: comportamiento idéntico.

  - Cero cambio de default.

### #2 — Usar `exec` en el `command` de `docker-compose` para que Django reciba SIGTERM

**Problema que Gemini ve:** con `sh -c "python manage.py migrate && python manage.py runserver"`, el shell queda como PID 1, no propaga señales al proceso Python, y `docker compose stop` espera los 10s default antes de matar con SIGKILL.

**Lo que pienso:**

- Gemini tiene **toda la razón técnicamente**. El problema es real y bien conocido.
- **Pero mirémoslo en contexto:** este es el comando del **servicio de dev**. En dev, `docker compose stop` lo más probable es que sea un `Ctrl+C` durante un `docker compose up` interactivo, o un `docker compose down`. En ambos casos el `runserver` se mata y Docker compose recrea el container en el próximo `up`. **No hay estado persistente en el proceso** — `runserver` no tiene nada que cerrar limpiamente.
- En producción, este **no es el comando** que se va a usar (el PR es explícito en que es dev). El target de prod es gunicorn/uvicorn, que sí maneja señales.
- La solución es **un cambio de una palabra** (`exec python`) y **no introduce ningún riesgo**. Es gratis.
- **Cumple tu regla DX:** "facilitar al máximo el trabajo de los desarrolladores pero sin intervenir el producto". Manejar señales de shutdown de forma más limpia es parte de la calidad del dev-loop (no del producto).

Esto **también vale la pena adoptarlo**.
