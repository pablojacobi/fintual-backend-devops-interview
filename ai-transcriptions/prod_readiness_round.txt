### Phase 3: structured logging

The `LOGGING` block in `core/settings_prod.py` is already in
place from Phase 0.5. I rebuild the prod image and check
`docker logs`. The gunicorn lines come out plain:

```
[2026-06-17 01:28:41 +0000] [1] [INFO] Starting gunicorn 26.0.0
```

Not JSON. I research and find that gunicorn has its own
`gunicorn.access` / `gunicorn.error` loggers, separate from
Django's `LOGGING`. The fix is `logconfig_dict` in
`gunicorn.conf.py`. I write one that routes both loggers
through `pythonjsonlogger.jsonlogger.JsonFormatter` and stdout.

I also enable `preload_app = True` so the SECRET_KEY /
ALLOWED_HOSTS guards in `core/settings_prod.py` fail at master
boot, not in a worker respawn loop. Without `preload_app`, each
worker crashes on its first request and gunicorn keeps
respawning workers in a tight loop — the operator sees logs but
the container never dies cleanly.

Rebuild + run:

```
$ docker run --rm -e DJANGO_SECRET_KEY=test -e DJANGO_DEBUG=0 \
    -e DJANGO_ALLOWED_HOSTS=localhost backend-devops:prod
  # ImproperlyConfigured at boot, container exits 1, no worker loop
```

With the right secret and a real DB:

```
# 28 lines of stdout, all valid JSON (asctime, levelname, name,
# message, h, r, s, b, D, f, a — atoms are null on non-access
# lines, which is fine)
$ docker logs <web> | python3 -c '...'  # OK: 28  Bad: 0
```

I commit Phase 3 as `feat(prod): route gunicorn's own loggers
through a JSON formatter`.
